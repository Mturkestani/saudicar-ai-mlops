Feature Engineering

Feature engineering is how we turn raw car listings into numbers a model can actually learn from. This page explains why models need numbers, what encoding is, why it matters, and how each step applies to our real Saudi used car dataset.

The core problem

Why does the model want numbers?

A machine learning model is, underneath, a big pile of math. To predict a price it multiplies each input by a weight and adds the results together, or it compares numeric values to split the data into groups. Every one of those operations — multiply, add, compare as greater/less — only works on numbers.

What a model can do

Multiply Mileage × weight. Ask is Year > 2015?. Add many numeric signals into one predicted price. All pure arithmetic.

What a model cannot do

It cannot multiply the word "Toyota". It cannot ask is "Camry" > "Patrol"?. Text has no numeric value, so the math simply has nowhere to start.

A linear model literally computes something like this:

# The model learns one weight per numeric input price = (w1 * car_age) + (w2 * Mileage) + (w3 * Engine_Size) + (w4 * is_make_toyota) # 0 or 1 + ... + bias
Key idea: Our dataset is full of text — Make, Type, Fuel_Type, Gear_Type, Region. The model can't touch any of it until we convert it into numbers. That conversion is the main job of Day 6.
Vocabulary

What is a “feature”?

A feature is one input signal the model uses to make its prediction. In our dataset each column (except the target) is a feature candidate. The target is the thing we want to predict: Price.

The heart of Day 6

What is encoding — and why it matters

Encoding means converting a text category into a number (or a set of numbers). It matters because a wrong encoding doesn't just fail — it can quietly lie to the model and hurt accuracy without any error message. Here are the three encodings you need, and exactly when to use each on our data.

1. One-Hot Encoding

Make one new 0/1 column per category. A row gets 1 in the column for its category and 0 everywhere else. Best for columns with few distinct values (low cardinality).

Gear_Type (raw)gear_Automaticgear_Manual
Automatic10
Manual01
Automatic10
Use on our data for: Gear_Type (2 values), Fuel_Type (~3 values), Region (~10 values), and Make (a few dozen). These stay a reasonable number of new columns.

2. Ordinal (Label) Encoding

Map each category to a single integer: 0, 1, 2…. This is only correct when the categories have a real order, because the numbers imply 2 > 1 > 0.

The classic trap: If we encode Make as Toyota = 0, Ford = 1, Lexus = 2, the model believes Lexus > Ford > Toyota as numbers, and even that Toyota + Toyota = Ford. That ordering is fake — brands have no natural order. This silently damages the model. So we do not ordinal-encode Make, Type, or Region.

Ordinal encoding would only make sense for something like small < medium < large. Our dataset doesn't really have such a column, so we mostly avoid it.

3. Target / Frequency Encoding

For columns with too many values to one-hot — our Type column (Camry, Patrol, Land Cruiser, Sonata… hundreds of models). One-hot would explode into hundreds of columns. Instead we replace each category with a single summary number:

Frequency encoding

Replace the category with how often it appears. Camry appears often → large number; a rare model → small number.

Target encoding

Replace the category with the average price of that category in the training data. “Land Cruiser” → its typical price.

Powerful but risky: target encoding uses the price to build a feature, so it can leak the answer if done wrong. See the leakage section below. For our first model, one-hot the easy columns and leave Type out or frequency-encode it.
Quick rule of thumb: few categories → one-hot. Real order → ordinal. Too many categories → frequency or (careful) target encoding. Never ordinal-encode something unordered like a brand.
Beyond encoding

Improving the numeric features

Year → car_age

Models learn “how old is the car” better than a raw year. We create car_age = 2026 - Year. A 2018 car becomes 8. This is a brand new feature we engineered — that's literally the name of the topic.

Price → log(Price)

Our EDA showed price is skewed: median 58,000 but max 1,150,000 SAR. Training on log(Price) tames the luxury outliers; we take exp() of the prediction to get SAR back.

Mileage & Engine_Size

Keep them, but the extreme values (a car with 20,000,000 km!) should be removed in Day 5 cleaning. Feature engineering assumes the input is already clean.

Optional: scaling

Linear models like inputs on a similar range. Mileage is in the hundreds of thousands; Engine_Size is ~2–6. A scaler can put them on a common scale. Tree models (Random Forest) don't need this.

Why these help: our EDA correlations were Year 0.308, Engine_Size 0.304, Mileage -0.119. Newer + bigger engine → higher price; more mileage → lower price. Good features make these real relationships easier for the model to see.
Interview gold

The one rule that matters most: no data leakage

Data leakage (تسريب البيانات) happens when information from the test set (or from the answer) sneaks into your features. The model looks amazing in your notebook and then falls apart in production. This is the single biggest thing that separates a “notebook model” from an MLOps engineer.

Wrong — leaks the answer

Compute “average price per Make” over the whole CSV, then use it as a feature. Rows you'll test on helped build the feature, so scores look unrealistically good.

→

Right — fit on train only

Split first. Learn every encoder / average / scaler from the training rows only, then apply those fixed rules to the test rows and to live requests.

The golden rule: fit encoders and scalers on the training data only, then transform everything else with them. Never let the test set influence how a feature is built.

Connecting to the product

Why we wrap this in a pipeline

The same transformation must run at training time and at prediction time in the API. If they differ even slightly, predictions go wrong. A scikit-learn Pipeline bundles “encode + scale + model” into one object we fit once and save as an artifact.

# The shape of what we'll build in src/saudi_car_ai/features/ build_features(df) # create car_age, pick columns preprocessor.fit(train_X) # learn encoders on TRAIN only X = preprocessor.transform(X) # apply same rules everywhere joblib.dump(preprocessor, ...) # save artifact for the API
Big picture: in our architecture the flow is Cleaning → Feature Engineering → data/processed → Training. The encoder we build here is later loaded by the FastAPI service so a live request like {"Make": "Toyota", "Year": 2019, ...} is transformed the exact same way as training data. Feature engineering is where the “model” quietly becomes a “product”.
Check yourself

Mini quiz

Try to answer before opening each one.

Why can't the model use the word “Toyota” directly?
Because the model is math — it multiplies, adds, and compares numbers. Text has no numeric value, so we must encode it (e.g. one-hot into make_Toyota = 1).
Gear_Type has 2 values, Type has 100+. Which encoding for each?
Gear_Type → one-hot (few values, cheap). Type → frequency or target encoding (one-hot would create hundreds of columns). Never ordinal-encode either, because their categories have no real order.
Why is car_age a better feature than Year?
Price depends on how old the car is. car_age = 2026 - Year expresses that directly, so the model learns the relationship more easily than from a raw year number like 2018.
I compute “average price per Make” over the whole CSV and use it as a feature. What's the mistake?
That's data leakage — test rows helped build the feature, so results look too good and fail in production. Fix: split first, then compute the averages on the training set only and apply them to the rest.
Why do we consider predicting log(Price) instead of Price?
Price is heavily skewed (median 58k, max 1.15M SAR). A few luxury cars distort training. Predicting log(Price) compresses that range; we exp() the prediction to return SAR.

Glossary

Feature

One input signal the model uses to predict, e.g. car_age or Mileage.
Target
الهدف
The value we predict. For us it's Price (or log(Price)).
Encoding
الترميز
Converting a text category into numbers so the model can use it.
One-hot encoding
One 0/1 column per category. Best for few categories.
Ordinal encoding
Category → single integer. Only for ordered categories.
Cardinality
How many distinct values a column has. Gear_Type is low; Type is high.
Data leakage
تسريب البيانات
Test/answer information leaking into features. Causes fake-good scores.
Pipeline
An object bundling preprocessing + model so the same steps run in training and in the API.
